Generate an HTML progress summary using the shared template at **`specs/_template/summary.html`**.

`$ARGUMENTS` is a space- (or comma-) separated list of feature slugs.

- **One slug** → single-feature mode. Write to `specs/<slug>/summary.html`.
- **Two or more slugs** → **epic mode**. The slugs form one epic; the **first slug is the umbrella** (its title/problem drives the hero, and the file is written to its folder). Write the combined page to **`specs/<first-slug>/summary.html`**.

The page summarises progress for a non-technical stakeholder — what it is, where it stands, what's been agreed (acceptance criteria), the domain knowledge it touches, and the proof it works (tests) — and **explicitly flags any artifact that was scanned for but not found**.

## Preconditions (check before starting)
- [ ] `specs/_template/summary.html` exists (the template this command fills in).
- [ ] For **every** slug passed: `specs/<slug>/` exists and `specs/<slug>/spec.md` exists.

If the template is missing, or any slug's folder/`spec.md` is missing, STOP and report exactly which slug is missing. Do not invent content. (If only some slugs are missing, report them and ask whether to proceed with the rest.)

## What to read — for EACH slug
Read every file that exists; skip silently the ones that don't, but **record which expected artifacts are missing** (they drive the At-a-glance audit table & the pipeline). Paths follow this branch's layout, with legacy fallbacks:

1. `specs/_template/summary.html` — **the template** (read once). Carries the `<style>` block, palette, header logo, the Overview section (`<section id="overview">`) which holds the intro + capabilities + the "At a glance" snapshot (roll-up, audit table, AgenticFX pipeline), the epic `<section id="slugs">` block, the `<section id="rules">` (Domain Knowledge: Problem→Solution · Personas · Glossary · Business rules — all tables), and the `<section id="tests">` (Test Results) block.
2. `specs/<slug>/spec.md` — title, problem statement (`## Problem`), persona line (`## User & outcome`), scope, status, dates, `## Related rules`, `## BR Conflicts`, AND the `## Acceptance criteria` section (every AC: id, text, pass/pending `[x]` vs `[ ]`). ACs live inline in the spec on this branch.
3. **Legacy fallback for ACs** — older/shipped slugs may keep ACs in a separate `specs/<slug>/acceptance-criteria.md`. If that file exists, read ACs from it; otherwise read them from `spec.md`'s `## Acceptance criteria` section. (Never read both — prefer the spec's inline section when present.)
4. **Coverage note** — read the top of `tests/cases/<slug>.md` (replaces the old `test-plan.md`); legacy slugs may still have `specs/<slug>/test-plan.md` — read it if present, else skip silently.
5. Test cases — search the **whole `tests/` root**: `tests/cases/<slug>.md` and recursively `tests/**/cases/**/<slug>.md` (covers `tests/joblogic-test-platform/cases/**/<slug>.md`); also worktree layout `worktree/**/tests/**/cases/<slug>.md`. Count + categorise (happy / edge / negative / regression). (None ⇒ Test Cases = missing.)
   - **Test scripts (automation)** — also scan for authored scripts per slug: `tests/e2e/<slug>.*` (e.g. `.spec.ts`) and `tests/integration/<slug>.*`, recursively under the `tests/` root and `worktree/**/`. Record which layers exist (e2e / integration / both / none) — this drives the **Test scripts** column. (None ⇒ Test Scripts = missing.)
6. Planning stage — slug lives as a **folder**: `planning/{backlog,active,done}/<slug>/` (read `item.md` / status log inside for dates). Legacy fallback: `planning/<stage>/<slug>.md`. Most-advanced wins (backlog → active → done, or spec status `shipped`). Note: the parent folder may be named differently (e.g. `planning/done/<Epic-Name>/`) — match on the slug file inside.
7. Test report / execution — search the **whole `tests/` root** for any of: `tests/reports/<slug>/report.md`, `tests/_runner/reports/<slug>/**/summary.md` (runner execution summary; the deepest timestamp folder is the latest run), `tests/**/reports/<slug>/report.md`, and `tests/e2e/<scope>/reports/<scope>/report.md` (and the same under `worktree/**/`). Parse **Verdict** (🟢 GREEN / 🔴 FAIL / BLOCKED / neutral), the result table (case / status / time), the **run date**, and coverage. If both a manual `report.md` and a runner `summary.md` exist for a slug, prefer the runner summary's verdict/run-date and merge their result rows. (None of these ⇒ Test Report = missing.)

Read once for the whole run (project-level, not per slug):
8. **Wave membership (epic mode)** — `planning/roadmap.md` and, if present, the umbrella slug's epic planning item (`planning/{backlog,active,done}/<epic>.md`, e.g. its **Breakdown** table). Map each slug passed to the **wave** it shipped in (a wave = a batch of child slugs delivered together, e.g. "v1 MVP", "v2 features", "visual-polish wave 1/2/3"). Use only what these files state — never invent a wave or guess membership. This drives the **Waves rollup** and the epic `{{WAVE_COUNT}}` stat. (Single-slug mode: skip — one slug is one wave, so the Waves block is deleted.)
9. `docs/domain/personas.md` — persona library. Pick the personas the spec(s) name in their `## User & outcome` line.
10. `docs/domain/glossary.md` — term library. Select **only** terms the spec(s)/test cases actually reference (grep the spec text for each glossary term); do not dump the whole file.
11. `.claude/metrics/usage-summary.json` — **optional**, git-ignored, produced by the Claude Code token-usage hook (`scripts/track-usage.py`, wired to Stop/SubagentStop/SessionEnd). Project-wide Claude Code token usage & **estimated** cost across all sessions in this repo. Read `totals` (`tokens`, `input`, `output`, `cache_write`, `cache_read`, `cost_usd`), `cost_usd_estimate`, `sessions`, and `by_model`. Token counts are exact; **cost is a list-price estimate** — always label it as such. If the file is absent, the build still proceeds: render the usage band as an `.empty-note` ("no usage metrics captured yet"). This is repo-wide instrumentation, **not** per-slug — show it once, never fabricated per feature.

## How to build the page
1. **Start from the template.** Read `specs/_template/summary.html` and use it as the exact starting point — the single source of styling truth.
2. **Pick the mode** (template's leading mode comment explains both). Both modes keep: **Overview (At-a-glance snapshot → AgenticFX pipeline → What it does) · Domain Knowledge · Test Results**.
   - **Single-slug:** fill Overview (incl. its at-a-glance snapshot: roll-up + a one-row audit table + the pipeline), Flow (optional), Acceptance Criteria, Timeline; Domain Knowledge uses **one** `.rules-group` (drop its `.rg-head` slug label); **delete `<section id="slugs">`**. Nav anchors, in scroll order: `#overview #ac #rules #tests #timeline`.
   - **Epic (2+ slugs):** hero = umbrella (first) slug's title/problem only (no stats band); the roll-up totals live in the Overview snapshot; fill `<section id="slugs">` with one `.slug-card` per slug **in the order passed** (umbrella first); Domain Knowledge **repeats** one `.rules-group` per slug (umbrella first); **delete** the single-slug detail sections (Flow, Acceptance Criteria, Timeline). Nav anchors, in scroll order: `#overview #slugs #rules #tests` (Domain Knowledge comes after Slugs).
3. **Fill placeholders & expand repeats.** Replace every `{{PLACEHOLDER}}`; duplicate each `<!-- REPEAT … -->` inner element per data item; delete the REPEAT markers.
4. **Drop empty parts honestly.** Missing artefact → name it in the audit table's **Gaps** cell, reflect it in the AgenticFX pipeline (skipped/unknown step), AND show the relevant section/block as an `.empty-note` / neutral "not started" / "no automated runs yet". Never fabricate counts, dates, verdicts, rules, personas, or glossary terms.
5. **Keep the chrome intact.** Leave the `<style>` block, the `:root` palette, the `<img class="logo" …>` JobLogic logo, and the scroll-spy `<script>` at the end of `<body>` exactly as in the template (the script highlights the nav link of the section in view; just make sure the nav anchors match the sections you keep).
6. **Strip the template's instructional comments.** Delete the leading `<!-- TEMPLATE for /gen-summary … -->` and the `<!-- TWO MODES … -->` comments from the generated file.

### Data → template mapping
- **Overview** (`<section id="overview">`): heading + stage badge, then titled (`dk-block-title`) blocks in this order — **(1) At a glance**: a `rollup` stat band (epic totals e.g. # slugs, **# waves** (epic mode, when the slugs span ≥1 named wave), ACs verified/total + %, test runs, business rules, shipped count) + a per-slug `audit-table` + a one-line missing-summary; **(2) Agentic build cost** (see next bullet); **(3) AgenticFX pipeline** (see "AgenticFX pipeline" bullet); **(4) What it does**: the intro (`{{OVERVIEW_SUBTITLE}}`) + the capabilities `features-grid`.
- **Agentic build cost** (Claude Code usage, from `.claude/metrics/usage-summary.json`; render as its own `dk-block-title` "Agentic build cost" + a `rollup` stat band — **reuse the existing `.rollup` component, add no new CSS**): show, as stat items, **Total tokens** (humanised, e.g. "739K"; this is `totals.tokens`), **Est. cost** (`$` + `cost_usd_estimate`, with a "list-price estimate" sub-label or a `data-tip`), **Sessions** (`sessions`), and an **Input / Output** split (`totals.input` / `totals.output`); optionally a **Models** line from `by_model` keys when more than one. Caption it as repo-wide Claude Code spend, **not** per feature. **Tokens are exact; cost is an estimate** — never present the dollar figure as a billed amount. If `usage-summary.json` is **absent or unreadable**, render the band as a single `.empty-note`: "No Claude Code usage metrics captured yet — the token-usage hook writes them after the next session." Never fabricate token or cost numbers.
- **AgenticFX pipeline** (`.pipeline`, one epic-level strip; it is the **second block** of Overview, after "At a glance" and before "What it does"): the steps are **Capture Idea · Spec · Implement · Unit Tests · Test Cases · Test Scripts · Test Execution · Ship** (no UI Design / Code Review). Set each `.pstep` class from evidence: `done` (✓ — artifact present / clearly complete), `skipped` (✗ — the artifact is missing for **all** slugs, i.e. bypassed), `partial` (~ — present for some slugs only, epic), or `unknown` (? — genuinely can't tell). Evidence map: Spec→spec.md; Test Cases→tests/**/cases; Test Scripts→e2e scripts; Test Execution→report.md; Ship→planning/done or status `shipped`; Capture Idea→a planning item exists. **Implement & Unit Tests** (no code artifact is scanned) are derived from **stage + planning status log**: slug `shipped` → `done`; `active` with the planning log showing implementation progress (e.g. "Tn DONE", "Implementation …") → `done` (or `partial` if clearly mid-way); otherwise `unknown`. Never invent code evidence. **No caption under the pipeline** — instead, every NON-done step (skipped / partial / unknown) carries a `data-tip="<why>"` that shows a hover tooltip explaining the gap; done steps get no `data-tip`. One table row per slug (umbrella first), columns **Slug · Stage (status badge with the date inside it, e.g. "✓ Shipped · 2026-06-16"; omit the date if none) · ACs (`{{AC_PCT}}` · verified/total) · Tests (verdict or "no runs") · Rules (`N rules · M conflicts`) · Gaps**. The **Gaps** cell is short free-text of what's missing for that slug — e.g. "report, cases" — or "none" (`class="gap none"`). Set `{{MISSING_SUMMARY}}` to a one-line tally below the table.
- **Stage badge** (`{{STAGE_CLASS}}` / `{{STAGE_LABEL}}`): `backlog` / "Backlog", `in-progress` / "In Progress", `shipped` / "✓ Shipped · <date>" — per slug.
- **Waves rollup** (`<section id="slugs">`, **epic mode only**, above the slug cards): one `.wave-row` per delivery wave, **in delivery order**, derived from `planning/roadmap.md` / the epic Breakdown table (read step 8). Each row: `{{WAVE_NAME}}` (e.g. "v1 MVP", "visual-polish wave 2"), `{{WAVE_SLUG_COUNT}}` (# of the passed slugs in that wave), `{{WAVE_SLUG_LIST}}` (their slug ids, comma-separated), and a `status-badge` whose `{{STAGE_CLASS}}`/`{{WAVE_STATUS_LABEL}}` is the wave's **rolled-up** stage — `shipped` only if EVERY slug in the wave is shipped, else `in-progress` if any is active, else `backlog`. Set `{{WAVE_COUNT}}` to the number of distinct waves. **If the slugs have no wave structure (or single-slug mode), DELETE the whole Waves block** (heading + `.waves`). Never invent a wave or guess membership — use only what roadmap/planning state.
- **Hero**: badge + title + slug line + one-line problem statement only — **no stats band** (the headline numbers live in the Overview roll-up, to avoid duplication).
- **Slug card** (epic): title + slug + stage badge + `sc-stat` mini-numbers + AC totals pills + scope `tag` chips. Keep compact.
- **Acceptance Criteria** (single section): group by the headings in the `## Acceptance criteria` section of `spec.md` (or the legacy `acceptance-criteria.md` if that's where they live); `ac-check pass` (✓) for `[x]`, `pending` (—) for `[ ]`.
- **Domain Knowledge** (`<section id="rules">`, both modes) — Problem/Personas/Glossary are all **tables** (`.dk-table`), shown once; then per-slug business rules. In order:
  - **Problem statement & solution** — a Problem→Solution table that **collects, never transforms**. Use **one row per problem exactly as the spec's `## Problem` presents them**: a single narrative / paragraph → **ONE row**; only use multiple rows when the spec itself enumerates multiple distinct problems (numbered / bulleted). **Never split a single statement into several.** Copy each problem's wording from the spec; the **Solution** column is taken from the spec's stated outcome/scope (`## User & outcome` / `## Scope`) where present, else left blank — do not invent it. Epic: use the umbrella spec.
  - **Personas** — table (Persona · Role & context · Success looks like), from `docs/domain/personas.md`, only those the spec's persona line names. None named → single row with an `.empty-note`.
  - **Glossary** — table (Term · Definition), only terms referenced by the spec(s)/tests, transcribed verbatim from `docs/domain/glossary.md`; set `{{GLOSSARY_COUNT}}`. No matches → single row with an `.empty-note`.
  - **Business rules & conflicts** — one `.rules-group` per slug (epic) or one (single): Related-rules `.rule-list` from `## Related rules`; BR-conflicts `.br-table` from `## BR Conflicts`. Empty → `.empty-note`. Never invent a rule or conflict.
- **Test Results** (`<section id="tests">`, both modes): TWO parts, in order — **(A) Test health at a glance** (aggregate, manager-facing), then **(B) Per-slug detail**.
  - **(A) Test health at a glance** — a `rollup` stat band (4–6 aggregate items) + a per-slug `audit-table` (`Slug · Test cases · Test scripts · Execution · Pass rate · Last run`) + a one-line `{{TEST_MISSING_SUMMARY}}` tally.
    - **Roll-up items** (aggregate across all slugs — pick the set that lets a manager grasp the ticket/epic at a glance; never fabricate, use `0`/`✗` when absent): **Verdict roll-up** (e.g. "4 🟢 · 1 no run"), **Case pass-rate** (passed/total across all result tables, + %), **Cases designed** (total designed cases), **Scripts authored** (slugs with ≥1 e2e/integration script, e.g. "4/5 slugs"), **Slugs executed** (slugs with a report/runner summary, e.g. "4/5"). Add a category split (happy/edge/negative/regression) only if it fits.
    - **Per-slug status row** (umbrella first): **Test cases** = `<n> designed` from the cases file or `✗ none` (`class="gap"`); **Test scripts** = layers found, e.g. `e2e + integration` / `integration only` / `✗ none`; **Execution** = `verdict` pill (`green`/`fail`/`neutral`, neutral "no runs" when none); **Pass rate** = `{{PASS_PCT}} · passed/total` (or `—` if no run); **Last run** = run date (or `—`).
  - **(B) Per-slug detail** — one `report-card` per slug under the "Per-slug detail" `<h3>`. If a report/runner summary exists, set the verdict pill (`green`/`fail`/`neutral`) and fill the `result-table` from its result rows; else `verdict neutral` + "no automated runs yet" note and delete the empty result table. (Per-slug case-category counts are already shown in the at-a-glance status table, so no separate category grid is rendered.)
  - The aggregate numbers in (A) MUST be consistent with the per-slug rows and the (B) cards — same verdicts, same counts.
- **Timeline** (single only): real dates only; `tl-dot done` for ship.

## Output rules
- Single-slug → `specs/<slug>/summary.html`. Epic → `specs/<first-slug>/summary.html`. Overwrite if it exists.
- One valid HTML5 file. The only external resource is the JobLogic logo URL already in the template; add no other external CSS/JS/fonts.
- Every number, date, AC, rule, persona, glossary term, and test result must come from files you read — **do not fabricate**. A scanned-but-absent artifact is `✗`, never a guess.

## Done criteria
- The `summary.html` exists, opens in a browser, styling identical to the template (palette, logo, layout).
- No `{{PLACEHOLDER}}` tokens and no `REPEAT` / `TEMPLATE` / `TWO MODES` comment markers remain.
- Correct mode: single has no `<section id="slugs">`; epic has slug cards and no leftover single-slug detail sections. Epic mode with a wave structure shows the **Waves rollup** (one lane per wave, correct `{{WAVE_COUNT}}`, rolled-up per-wave status) above the slug cards; single-slug mode (or no wave structure) has the Waves block deleted. Both modes have the Overview section (intro + capabilities + at-a-glance snapshot + pipeline), Domain Knowledge (Problem→Solution + Personas + Glossary + rules — all tables), and Test Results (the "Test health at a glance" aggregate block — roll-up + per-slug status table covering cases/scripts/execution — BEFORE the "Per-slug detail" report-cards).
- The Test Results aggregate numbers (verdict roll-up, pass-rate, cases/scripts/executed counts) are consistent with the per-slug status rows and the per-slug detail cards — no contradictions.
- The Overview audit table's Gaps cell honestly lists every missing artifact for each slug (or "none"); the roll-up numbers match the per-slug rows; the pipeline marks skipped/unknown steps with hover tooltips.
- The **Agentic build cost** band is present in Overview: either real numbers transcribed from `.claude/metrics/usage-summary.json` (tokens exact, cost labelled as an estimate) or a single `.empty-note` when the file is absent — never fabricated.
- Tell the user the file path, a one-line content summary, AND a one-line list of missing artifacts — e.g. "4 slugs, 57/64 ACs verified, stage: Shipped, ~$3.33 Claude Code spend; missing: test reports (all 4), test cases (all 4)".
