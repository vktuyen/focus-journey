# Acceptance criteria

Each item is a checkable, observable statement. If it isn't testable, rewrite it.

Scope reminder: this slice is the **pure, framework-free `JourneyEngine`** — a *domain* core loop driven by `tick(delta)`, an **injected clock**, and an **injected `ActivityPlugin`** (the shipped `activity-detection` interface). Every AC below is observable against the engine in isolation: assert on its exposed values (`distanceKm`, `activeTimeToday`, `rawActiveTime`, `idleTimeToday`, `state`, `mode`) after feeding a deterministic tick sequence. **No real timers, no `DateTime.now()` inside the engine, no Flame, no native code, no UI.** The engine *consumes* raw signals from `ActivityPlugin` — `getSystemIdleSeconds()` + `isScreenLocked()` only; **there is no sleep boolean**, so sleep is *inferred* from a **large idle reading** (or lock) — **not** from a large tick `delta` alone, which is instead clamped to `maxTickDelta` (Kevin ratified 2026-06-23, review M-1).

**Status (2026-06-23):** all 21 ACs verified — implementation `approved` (`/review-code`) and tests `green` 63/63 (`/execute-tests`, report `tests/_runner/reports/journey-engine/20260623-181042/`); all TC-001…TC-022 map to passing tests. **Review finding M-1 ratified by Kevin (2026-06-23):** sleep is inferred from a **large idle** reading only (or lock) — a large `delta` **alone** is not sleep but is clamped to `maxTickDelta` and still credited, since a stalled ticker must not silently delete genuine active time. AC-5/AC-6 and TC-007 reworded to match; AC-8's both-large case is unaffected (and tested). M-2 (`tickFromPlugin` error policy) and minor test-hardening remain as non-blocking follow-ups.

Decision model (Kevin, 2026-06-23): **two independent knobs** — grace window `G` and idle threshold `T`, `G ≤ T`, default `G = T = 5 min`. Bands by true-idle elapsed `s` (active floor `F`): `s ≤ F` → **active**; `F < s ≤ G` → **grace/travelling**; `G < s ≤ T` → **idle** (stopped); `s > T` or locked or sleep-inferred → **paused**. Distance + journey time accrue in active + grace only.

- [x] **AC-1 (P0): Active state accrues distance via speed-only `kmPerActiveHour`.** **Given** the engine is configured with a single shared `kmPerActiveHour` and the mock `ActivityPlugin` reports idle-seconds below the threshold, screen unlocked, and not sleeping, **when** a `tick(delta)` advances the engine while in the **active** state, **then** `distanceKm` increases by exactly `kmPerActiveHour × (delta in hours)` (within floating-point tolerance) and `state == active`. _(Depends on the `kmPerActiveHour` seam open question — written against the recommended default that the engine takes the rate as injected config/constant, with the value owned by `route-progress` province data; review must confirm the seam so both slices agree.)_
- [x] **AC-2 (P0): Journey time and raw active time are tracked as two separate accumulators, never conflated.** **Given** a tick sequence containing both genuine input (idle below threshold) and grace-window ticks (true idle within the grace), **when** the sequence is fed to the engine, **then** `activeTimeToday` (journey time) accrues for **both** the genuine-input and the grace ticks, while `rawActiveTime` accrues **only** for the genuine-input ticks — so after any mixed sequence `rawActiveTime ≤ activeTimeToday`, and the two are never equal unless zero grace was consumed. (Headline honesty rule.)
- [x] **AC-3 (P0): Active accrues distance + journey time + raw active time when input is fresh AND unlocked.** **Given** the mock reports idle-seconds at/below the small **active floor `F`** (genuine recent input) and screen unlocked, **when** a tick advances the engine, **then** `state == active` and that tick's `delta` adds to `distanceKm`, `activeTimeToday`, **and** `rawActiveTime` (and nothing adds to `idleTimeToday`).
- [x] **AC-4 (P0): Grace window counts as journey time + distance but NOT raw active time.** **Given** the user has gone idle and the elapsed true-idle `s` is in the **grace band** `F < s ≤ G` (screen unlocked, not sleep-inferred), **when** a tick advances the engine inside that grace, **then** `state` remains travelling, `distanceKm` and `activeTimeToday` accrue for that tick, but `rawActiveTime` does **not**.
- [x] **AC-5 (P0): Past the grace window (or locked, or sleep-inferred) → no travel; only idle time accrues.** **Given** any one of: true idle `s` has exceeded the **grace window `G`**, the mock reports screen **locked**, or sleep is inferred from a **large idle** reading (a large `delta` **alone** is **not** sleep — an over-sized travelling tick is clamped to `maxTickDelta`, see AC-8), **when** a tick advances the engine, **then** `distanceKm`, `activeTimeToday`, and `rawActiveTime` do **not** change, and only `idleTimeToday` accrues for that tick.
- [x] **AC-6 (P0): Screen-lock / sleep-inferred overrides the grace window.** **Given** the user has just gone idle and `s` is **within** the grace band `F < s ≤ G` (which alone would still count as travel), **when** a tick arrives with the mock reporting screen **locked** (or sleep inferred from a **large idle** reading — **not** from `delta` alone), **then** the engine treats it as non-travel immediately — `distanceKm` and `activeTimeToday` do **not** accrue for that tick — i.e. lock/sleep wins over grace.
- [x] **AC-16 (P1): Threshold `T` distinguishes `idle` from `paused` state beyond the grace.** **Given** two independent knobs with `G < T` (so the band `G < s ≤ T` is non-empty), **when** true idle `s` falls in `G < s ≤ T`, **then** `state == idle` (vehicle stopped, no distance), and **when** `s > T` (or locked / sleep-inferred), **then** `state == paused` — in **both** bands the accounting is identical (only `idleTimeToday` accrues; no distance/journey/raw). _(Confirms the proposed `[G, T]` middle-band semantics — spec open question; review must confirm distance stops at `G`, not `T`.)_
- [x] **AC-7 (P0): Elapsed is computed from the supplied delta (last-tick timestamp), never an assumed fixed interval.** **Given** two tick sequences that reach the same total elapsed time but with different per-tick `delta`s (e.g. one 60 s tick vs six 10 s ticks, all active), **when** each sequence is fed to the engine, **then** the resulting `distanceKm` and `activeTimeToday` are equal (within tolerance) — the engine scales by the caller-supplied `delta` and never assumes a constant tick period, and reads no clock of its own to fill gaps.
- [x] **AC-8 (P0): A sleep/wake gap counts as neither journey nor active time.** **Given** the engine was active, then a long real-time gap passes (machine asleep) and the next `tick(delta)` arrives with a **large** `delta` and the mock reporting a **large idle** value (sleep inferred — there is no sleep flag), **when** that tick is processed, **then** the gap is attributed to **idle** — `distanceKm`, `activeTimeToday`, and `rawActiveTime` do **not** increase for the gap, and it is not silently accrued as travel. (Consumes the shipped `activity-detection` AC-9 "large idle after wake"; this engine interprets that signal as idle.)
- [x] **AC-9 (P1): Daily counters reset at local midnight while cumulative distance persists.** **Given** the engine holds non-zero `activeTimeToday`, `rawActiveTime`, `idleTimeToday`, and `distanceKm` for a given local calendar day, **when** the injected clock crosses **local midnight** and the next tick is processed, **then** `activeTimeToday`, `rawActiveTime`, and `idleTimeToday` reset to zero while cumulative `distanceKm` (position) is **preserved** unchanged.
- [x] **AC-10 (P1): App-closed-across-midnight is detected from the stored date on restore.** **Given** persisted state whose **stored calendar date is earlier than the injected clock's current local date**, **when** the engine restores that state, **then** the daily counters (`activeTimeToday`, `rawActiveTime`, `idleTimeToday`) are reset to zero for the new day while cumulative `distanceKm` is preserved, and the missed day is **not** reconstructed. _(Resolved — Kevin 2026-06-23: reset daily counters, do not reconstruct.)_
- [x] **AC-11 (P1): Persistence save/restore round-trip within the same local day resumes exactly.** **Given** the engine has accrued `distanceKm` and the day's counters and is saved via the repository seam (`shared_preferences`/JSON), **when** a fresh engine restores that state with the injected clock still on the **same local day**, **then** `distanceKm`, `activeTimeToday`, `rawActiveTime`, `idleTimeToday`, `state`, `mode`, and the stored date all equal the saved values — and a subsequent active tick continues accruing distance from the restored position (no double-count, no reset).
- [x] **AC-12 (P0): Determinism — identical tick sequences produce identical results with no wall-clock dependence.** **Given** the same injected clock script and the same mock `ActivityPlugin` value sequence, **when** the engine is run twice (or in two processes / at two real-world wall-clock times), **then** every exposed value (`distanceKm`, `activeTimeToday`, `rawActiveTime`, `idleTimeToday`, `state`, `mode`) is byte-for-byte identical across runs — the engine's output depends only on injected inputs, never on real elapsed wall-clock time.
- [x] **AC-13 (P1): `mode` is cosmetic only — same speed for all modes in v1.** **Given** two otherwise-identical active tick sequences that differ only in the engine's `mode` (travel skin), **when** each is fed to the engine, **then** the resulting `distanceKm` and time accumulators are equal — `mode` is preserved and exposed but does **not** affect `kmPerActiveHour` or any accrual in v1. (Per-mode speeds/energy are v2 `journey-energy-model`.)
- [x] **AC-14 (P1): Grace minutes consumed before exceeding the threshold stay attributed as travel.** **Given** the user goes idle, accrues some grace-window ticks (counted as journey time + distance), and then **exceeds** the idle threshold without returning, **when** the threshold-crossing tick is processed, **then** the engine does **not** retroactively reclassify the earlier grace ticks as idle — the grace seconds remain `distanceKm` + `activeTimeToday`, and only subsequent ticks accrue `idleTimeToday`. _(Resolved — Kevin 2026-06-23: grace stays travel; no retroactive rollback.)_
- [x] **AC-15 (P1): Streak-qualifying metric is exposed as raw active time.** **Given** downstream `local-stats` needs to decide whether a day qualifies for a streak (threshold ≥25 min), **when** it reads the day's qualifying duration from the engine, **then** the value exposed for streak qualification is `rawActiveTime` (true input, no grace), distinct from and ≤ `activeTimeToday`. _(Resolved — Kevin 2026-06-23: streak qualifies on raw active time. This slice only **exposes** the metric; streak counting itself is `local-stats`.)_

## Non-functional
- [x] **Determinism & testability (P0, headline):** The engine is fully unit-testable with **no real timers, no real idle waits, no `DateTime.now()` internally, and no native/Flame/UI dependency.** Clock and `ActivityPlugin` are constructor-injected behind interfaces; identical injected inputs yield identical outputs. (See AC-7, AC-12.)
- [x] **Framework-free purity (P0):** The engine imports no `flutter`, `flame`, or `dart:async` real `Timer`; it is plain Dart in the *domain* layer. Persistence is reached only through a repository interface (the `shared_preferences`/JSON implementation lives in the *data* layer, not the engine). Swapping real ↔ mock `ActivityPlugin` or real ↔ test clock requires no change to engine code.
- [x] **Privacy by construction (P0):** The engine only ever observes an **aggregate idle duration + lock/sleep boolean** via `ActivityPlugin` — never input content, keystrokes, screen, clipboard, files, mouse coordinates, or window titles. It introduces no dependency that widens that surface, and persists only aggregate counters + position + date (no raw signals). Re-checkable by `privacy-guardian` (`/privacy-audit`).
- [x] **Honest accounting (P0):** Across every code path, journey time (incl. grace) and raw active time remain two separate accumulators; no path conflates them or lets `rawActiveTime` exceed `activeTimeToday`. (See AC-2.)
- [x] **Robustness to gaps (P1):** Timer drift, missed ticks, and sleep/wake gaps are handled purely from the supplied `delta` + signal — a missing or oversized tick never silently accrues travel. (See AC-7, AC-8.)

## Out of scope (reminder)
- **Raw signal acquisition** — reading OS idle-seconds / lock / sleep state. That is the shipped `activity-detection` slice; this engine *consumes* its `ActivityPlugin` interface and does not acquire signals.
- **The periodic driver / real ticker service** — the app-layer service that runs a real `Timer`, computes `now − lastTick`, and calls `engine.tick()`. The engine is tested by feeding deltas directly; the ticker is app-layer wiring, not the pure engine.
- **Province chain / start-province / direction / map / "% of country"** — the engine emits a scalar `distanceKm`; mapping it onto the chain is `route-progress`.
- **Stats UI, streaks/badges, settings screen, onboarding** — that is `local-stats`. The engine only **exposes** the honest `rawActiveTime` those features consume (AC-15); it does not count streaks or render anything.
- **Flame scene / any rendering** — that is `journey-view`.
- **Per-mode speeds / energy / fuel model** — v2 `journey-energy-model`. v1 is speed-only with one shared `kmPerActiveHour`; `mode` is a cosmetic skin (AC-13).
- **Route-completion celebration / summary** — depends on the chain, lives with `route-progress`.
