What it does

Twenty slugs across four waves build one product: real focus time drives a virtual road trip up Vietnam. The engine is the honest core; everything else — scene, cockpit, map, stats, and the latest UI/UX-refinement wave — is a faithful view of it.

🧭
Honest core loop
journey-engine — pure framework-free Dart converts active focus into km (speed-only), tracking journey-time vs raw-active-time separately and persisting across sleep/wake & midnight.
🔒
Privacy-clean signal
activity-detection — reads only aggregate system idle-seconds + screen-lock (macOS Swift / Windows C++), never keystrokes or screen. The auditable trust foundation.
🛣️
POV road scene
journey-view → journey-scene-v2 → journey-scene-art-v3 — a Flame first-person road that winds, scrolls slower & calmer, and wears a cohesive hi-res stylized-flat Vietnam art set.
🚗
First-person cockpit
journey-pov + journey-cockpit-lean — a car / motorbike cockpit composited over the road that leans into the curves, bounded & reduce-motion-safe.
🗺️
Real-Vietnam map
route-progress + map-experience — an ordered province chain on real lat/long with OSM tiles, folded into the journey tab, with idle stretches painted red.
🧳
Route authoring
route-planner-v2 — pick any start & end province, mark stops (auto-inserted), review before starting, and abandon a route without ever resetting lifetime distance.
📊
Stats, badges & trust
local-stats — daily/weekly stats with raw-active shown separately from journey time, four badge families, and the onboarding/privacy screen whose claims must match the code.
🪟
Always-on-top presence
mini-window — a frameless always-on-top PiP compact mode plus a tray/menu-bar icon that keeps tracking after the main window is closed.
⏸️
Idle accounting
idle-accounting — counts idle from the moment the state changes (not the next tick) and produces the distance-keyed segment record the red map trace consumes.
🏍️
Choose your vehicle
vehicle-picker — a persisted icon picker (Settings + route-start) that cosmetically overrides the displayed skin (vehiclePreference ?? engineMode) without ever touching accrual.
🌀
Dynamic sweeping curve
journey-dynamic-curve — intensifies the road into F1-style bends that sweep with the scroll phase (no wall-clock), preserving even arc-length object spacing.
🇻🇳
Accurate Vietnam (2026)
vietnam-map-fidelity + province-chain-2026 — a bundled, always-offline base map of Vietnam plus the journey rebuilt onto the current 34 provinces (2025 reform), so the map and the route agree about what Vietnam is.
🧭
Real-road route & clean reset
route-real-road + journey-reset — the line follows a real bundled national highway (OSM/ODbL, offline) with big pins only on start/end/stops, and an explicit Start-over / Reset-everything flow that never silently loses lifetime progress.

Problem statement & solution — from the umbrella spec, journey-engine
ProblemSolution
Vietnam Focus Journey turns real focus time into a virtual road trip up/down Vietnam's province chain. Something has to be the core loop that converts "the user is actively working" into "the traveller moved N kilometres" — and does it honestly, deterministically, and in a way the rest of the app (Flame scene, map, stats) can trust. Today there is nothing between the raw OS signal and what the user sees: distance can't accrue, daily progress can't persist across restarts, and there is no separation between journey time (drives distance, includes the idle grace) and raw active time (drives stats/streaks) — the single most important honesty rule of the product. Deliver that loop as pure, framework-free Dart: an engine taking an injected clock + injected ActivityPlugin, advancing on real elapsed-time deltas, surviving sleep/wake gaps, and persisting/restoring daily progress locally. Exposes distanceKm, journey time, raw active time, idle time, state and cosmetic mode; two-knob active/idle/paused policy; speed-only distance; day-boundary reset at local midnight. Fully unit-testable with no real timers, Flame, or native code.

Retrospective — what to improve

An honest read of the agentic process that built these 20 slugs — where it costs speed and quality, and the highest-payoff fixes. Numbers are from the repo's own usage telemetry and artifacts.

One-line diagnosis: a big-team audit process is running a solo project — it optimises for traceability and ceremony over throughput and depth. The build works and ships green, but each feature costs ~$55 and moves through six manual phases, and "green" hides 21 acceptance criteria that were never actually verified.

🔻 Cons & downsides (why it's slow / average)

Context is re-read, not reused.Six phases, each a separate session whose sub-agents re-read the spec + architecture + domain docs cold. For every token of real output, 171 were reprocessed.cache-read : output = 171 : 1
Six sequential manual gates per feature.capture → new-feature → implement → review → test → ship. Only spec-signoff and green+review add safety; the other four session boundaries add wall-clock + cost.~$55 / slug · 6 commands each
Verification is shallow-but-late; volume masks rigor.One review pass, no adversarial multi-lens or property-based tests on the pure domain, and many ACs deferred to "manual on-device" then forgotten.21 / 330 ACs never verified
Feedback arrives at the end → expensive rework.route-real-road alone had 2 wrong route-model iterations, a phantom marker bug, and 6 review fixes applied after build — the human only sees results in Phase 3+.1 session spiked to $322
Ceremony inflation.Prose-heavy ledgers that duplicate git history, 9 ADRs for a solo desktop app, and ~1,200 lines of process instructions to hand-maintain (itself a drift source).test:code LOC = 1.8 : 1
No parallelism.Independent slugs in a wave run strictly serially through all six phases.

🔼 How to improve (ranked by payoff)

① Keep context warm.Run implement → review → test in one session, or hand sub-agents a pre-digested context pack (ACs + the 3–5 relevant files) instead of "go re-read docs/".target ~3–5× cheaper & faster
② Collapse phases for the common case.One orchestrator, pausing at exactly two human checkpoints (spec approved, ship). Widen the proven /quick-change lane.
③ Add a cheap design/preview checkpoint before full build.A mock or spike the human reacts to in minutes — kills the late-feedback rework, especially for UI/UX.
④ Trade test volume for targeted verification.Property-based tests on the pure domain, adversarial multi-lens review, and either close the 21 deferred on-device ACs or stop calling them ACs.
⑤ Trim the process surface.Status = one line + git link, not prose essays. ADRs only for cross-cutting calls. Consolidate overlapping commands/agents.
⑥ Parallelise independent slugs & right-size effort.Git worktrees for concurrent slugs; a cheap/fast model for mechanical work, deep reasoning reserved for design + review.
