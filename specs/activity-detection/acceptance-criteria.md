# Acceptance criteria

Each item is a checkable, observable statement. If it isn't testable, rewrite it.

Scope reminder: this slice exposes **raw signals only** (`getSystemIdleSeconds()`, `isScreenLocked()`, a mock source). It does not judge active vs idle, apply any threshold/grace, compute distance, pause/resume a journey, or render UI. ACs below are written against the `ActivityPlugin` interface, not any specific native implementation.

- [x] **AC-1 (P0): Idle seconds climb while untouched — macOS.** _(macOS automated smoke — `activity_logging_harness_test.dart` logged idle climbing 23s→47s on a real device; full manual idle-climb walk deferred per L3.)_ Given a macOS build using the real `ActivityPlugin` backend, when the machine receives no real user input for a period, then successive calls to `getSystemIdleSeconds()` return a monotonically non-decreasing value that reflects approximately the elapsed untouched wall-clock seconds (within a small tolerance).
- [ ] **AC-2 (P0): Idle seconds climb while untouched — Windows.** _(DEFERRED — see Known limitations L3: no Windows device, build MSVC-blocked; required before any Windows release.)_ Given a Windows build using the real `ActivityPlugin` backend, when the machine receives no real user input for a period, then successive calls to `getSystemIdleSeconds()` return a monotonically non-decreasing value approximating the elapsed untouched seconds (within a small tolerance).
- [ ] **AC-3 (P0): Idle resets to ~0 on real input — both OSes.** _(DEFERRED — manual checklist not executed; see L3.)_ Given idle seconds have been climbing on either macOS or Windows, when the user generates real input (key press or mouse movement/click), then the next call to `getSystemIdleSeconds()` returns a value at or near 0 (within a small tolerance reflecting the time since that input).
- [x] **AC-4 (P0): Screen-lock state is reported truthfully — macOS.** _(macOS automated smoke confirmed live read-shape `locked=false`; full manual lock/unlock walk deferred per L3.)_ Given a macOS build using the real backend, when the screen is locked, `isScreenLocked()` returns `true`; and when the screen is unlocked, `isScreenLocked()` returns `false`, reflecting the current OS lock state at call time.
- [ ] **AC-5 (P0): Screen-lock state is reported truthfully — Windows.** _(DEFERRED — see Known limitations L3 (no Windows run) and L1 (cold-start caveat).)_ Given a Windows build using the real backend, when the workstation is locked, `isScreenLocked()` returns `true`; and when unlocked, `isScreenLocked()` returns `false`, reflecting the current OS lock state at call time. _(Accepted limitation L1 — see Known limitations: a process **launched while already locked** reports `false` until the first lock/unlock transition.)_
- [x] **AC-6 (P0): Mock source is injectable and deterministic.** _(green automation — `mock_activity_source_test.dart`, `activity_flag_di_test.dart`.)_ Given the app/tests select the mock source (e.g. via `--mock-activity` or direct injection), when a caller sets the idle-seconds and lock values on the mock and then calls `getSystemIdleSeconds()` / `isScreenLocked()`, then the plugin returns exactly the caller-driven values, with no real OS access, no real timers, and no real idle waits — so results are fully deterministic and the swap between real and mock backends requires no change to calling code.
- [x] **AC-7 (P0): Privacy — only aggregate idle + lock are read.** _(`/privacy-audit` PASS — `privacy-guardian`, 2026-06-23.)_ Given a `privacy-guardian` audit of all `ActivityPlugin` code (Dart + native backends + mock), when the audit inspects the data the code reads or stores, then it confirms the code accesses ONLY an aggregate system-idle duration and the screen-lock boolean, and accesses NONE of: keystrokes, key contents, screen/display contents, clipboard, files, mouse-position history/coordinates, or window titles. The audit passes.
- [x] **AC-8 (P0): Privacy — no disqualifying new dependency.** _(`/privacy-audit` PASS; `screen_retriever` is transitive via `window_manager`, reads display geometry only — risk-cleared.)_ Given the dependencies introduced by this slice (any pub.dev package adopted from the spike, plus native libraries), when `privacy-guardian` reviews them, then no added dependency is capable of capturing input content, screen, clipboard, files, mouse-position history, or window titles; any such dependency is rejected.
- [ ] **AC-9 (P1): Large idle reported after a sleep/wake cycle.** _(DEFERRED — manual checklist not executed; see L3.)_ Given the machine has been asleep and is then woken, when `getSystemIdleSeconds()` is called shortly after wake (before any new input), then it returns a large value (not 0) consistent with the elapsed sleep duration — achieved via the standard OS idle API with no dedicated sleep/wake handling code in this slice. (Downstream `journey-engine` interprets this; this slice only guarantees the value is large, not 0.)
- [x] **AC-10 (P1): Graceful failure when the native signal is unavailable.** _(green automation — `method_channel_activity_plugin_test.dart`; typed `ActivityPluginException`, never `_TypeError`.)_ Given the underlying OS API is unavailable or permission is denied, when `getSystemIdleSeconds()` or `isScreenLocked()` is called, then the plugin surfaces the condition via a typed failure (a thrown `ActivityPluginException` / `Future` error) rather than crashing the process or returning a silently wrong value, and the error distinguishes "unavailable/denied" from a normal reading. (Chosen contract: surface an explicit error; the caller — `journey-engine` — decides the fallback policy.)
- [x] **AC-11 (P1): Observable contract is implementation-independent.** _(green automation — `activity_plugin_contract_test.dart` parametrized over both impls; native-equivalence portion bounded by L3.)_ Given the spike resolves to either an existing pub.dev package or a custom native plugin, when either implementation is exercised through the `ActivityPlugin` interface, then AC-1 through AC-5, AC-9, and AC-10 hold identically — the observable contract does not depend on which implementation was chosen.

## Non-functional
- [x] **Privacy (P0, headline):** _(`/privacy-audit` PASS.)_ The implementation reads only aggregate idle time and lock state; it never reads, buffers, logs, or persists input content, screen, clipboard, files, mouse coordinates/history, or window titles. Verifiable by `privacy-guardian` (`/privacy-audit`) and re-checkable on every dependency change. (See AC-7, AC-8.)
- [ ] **Cross-platform parity (P0):** _(DEFERRED — Windows unverified at runtime; see L3.)_ Both macOS and Windows implement BOTH `getSystemIdleSeconds()` and `isScreenLocked()` with equivalent observable behaviour. Neither capability may be macOS-only or Windows-only for v1.
- [x] **Performance (P1):** _(verified by code review — single synchronous reads, no internal polling/sleeping.)_ A single `getSystemIdleSeconds()` / `isScreenLocked()` call is cheap and non-blocking — it returns a current reading without sleeping/polling internally and without blocking the Dart UI isolate (suitable for repeated polling by the engine on a tick).
- [x] **Testability (P0):** _(green automation — deterministic mock, 36/36 with no real timers/OS.)_ The interface is injectable; the mock source enables fully deterministic unit tests with no real timers, no real idle waits, and no native/OS dependency, so `journey-engine` and tests can drive any signal value synchronously. (See AC-6.)
- [x] **Portability of contract (P1):** _(green automation — contract test over both impls.)_ The Dart `ActivityPlugin` interface is a domain contract; choosing or swapping the underlying native implementation or pub.dev package does not change its signature or observable behaviour. (See AC-11.)

## Known limitations (accepted at ship — 2026-06-23)
Surfaced by `/review-code` (`flutter-code-reviewer`); reviewed and **accepted as scope limits for this slice** rather than fixed. Tracked for a follow-up slug if real-world use demands it.

- **L1 — Windows cold-start lock state (review m1).** On Windows, lock state is cached from `WTS_SESSION_LOCK/UNLOCK` notifications and initialised to `false`. A process **launched while the workstation is already locked** (e.g. launch-at-login on a locked machine, or a crash-restart) reports `isScreenLocked() == false` until the next lock/unlock transition. macOS reads live and is unaffected. Impact is low: the downstream `journey-engine` also gates on idle-seconds, which climb correctly regardless. Fix direction if revisited: query initial state at registration via `WTSQuerySessionInformation`. Affects AC-5 / TC-007 / TC-008 (manual, Windows). Owner if reopened: `flutter-native-plugin-engineer`.
- **L2 — Negative idle coercion (review m3).** `_coerceIdleSeconds` accepts a negative `num` from the native channel and passes it through (e.g. `-1.0 → -1`) rather than treating it as `unavailable`. The real backends never emit a negative (macOS guards `>= 0`; Windows uses unsigned math), so this is only reachable via a faulty/hostile native return. Fix direction if revisited: reject negatives in `_coerceIdleSeconds` and add a unit case. Relates to AC-1 / AC-10. Owners if reopened: `flutter-app-developer` + `unit-test-writer`.
- **L3 — Windows real-OS verification deferred + macOS manual checklist not executed (ship decision, 2026-06-23).** Shipped **macOS-verified only**. The Windows native backend was authored and code-reviewed + privacy-audited, but never **run** on a Windows machine (no Windows device available; the build was MSVC-blocked on the dev Mac). Consequently **AC-2 (Windows idle), AC-5 (Windows lock)** and the **Cross-platform parity** NFR are unverified, and the 6-item real-OS manual checklist (`tests/cases/activity-detection-manual-checklist.md`) was not executed — so **AC-3 (input reset)** and **AC-9 (sleep/wake)** are also unverified by manual run (AC-1/AC-4 are backed by macOS automated smoke only, not the manual lock/unlock + idle-climb walk). **Required follow-up before any Windows release:** build + run on Windows hardware and execute the manual checklist on both OSes; only then check AC-2/AC-3/AC-5/AC-9 and the parity NFR. Owner: `flutter-native-plugin-engineer` + `test-executor`.

## Out of scope (reminder)
- The active/idle **decision** — applying any threshold (e.g. the 5-minute rule), grace window, or delta-from-last-tick. That is `journey-engine`.
- Pausing/resuming a journey, computing virtual distance, province-chain position, or any travel/pacing logic.
- Any UI, Flame scene, tray/menu-bar, mini-window, notifications, stats, or persistence.
- Sleep/wake *handling* logic (reacting to a large idle value on wake). This slice only guarantees the raw idle value is large after wake (AC-9).
- A dedicated sleep/wake signal or event — not added in this slice (revisit only if an OS misbehaves).
